# Предложения по улучшению приложения

Документ с идеями по доработке семейного приложения (Покупки / Задачи / Календарь / Профиль). Сгруппировано по приоритету — сверху то, что даст наибольший эффект при небольших затратах.

## Уже исправлено в этой итерации

Эти правки готовы (проверены `tsc` и `eslint` без ошибок) и ждут применения в репозиторий:

1. **Вступление в семью по коду** — раньше не работало: запрос к таблице `families` блокировался политикой RLS для тех, кто ещё не в семье. Заменено на RPC-функцию `join_family(p_code)` с `SECURITY DEFINER`, которая находит семью по коду и добавляет пользователя, не открывая чужие семьи на чтение.
2. **Политики RLS** — заменено устаревшее `auth.role() = 'authenticated'` (возвращает NULL в новых версиях Supabase) на `auth.uid() IS NOT NULL`; исправлены политики `SELECT`/`INSERT` для `families`; добавлены `DROP POLICY IF EXISTS` для идемпотентного применения.
3. **Адаптивный дизайн (телефон + ПК)** — на десктопе добавлена верхняя панель-навигация по разделам; точки-навигация справа теперь показываются только на телефоне (`md:hidden`).
4. **Чистка предупреждений линтера** — устранены ошибки `react-hooks/set-state-in-effect`.

## Высокий приоритет

**Офлайн-режим и оптимистичные обновления.** Список покупок — это то, что открывают в магазине, где связь часто плохая. Сейчас каждое действие ждёт ответа сервера. Стоит добавить оптимистичный UI (сразу показывать изменение, откатывать при ошибке) и кэш на стороне клиента, чтобы список открывался мгновенно.

**Реалтайм-синхронизация.** Семья пользуется списком одновременно: один отмечает «куплено», второй должен видеть это сразу. Supabase поддерживает Realtime-подписки на изменения таблиц — стоит подписаться на `shopping_items`, `tasks`, события календаря.

**Обработка ошибок для пользователя.** Сейчас при сбое запроса часто либо тишина, либо «не удалось». Нужны понятные сообщения и повтор действия (retry), особенно для сетевых ошибок.

**Подтверждение удаления.** Удаление магазина/задачи/события без подтверждения легко сделать случайно. Добавить диалог подтверждения или «Отменить» (undo) на несколько секунд.

## Средний приоритет

**Уведомления.** Push или email-напоминания о событиях календаря и задачах с дедлайном. Это превращает приложение из «списка» в реально полезный семейный органайзер.

**Назначение задач и ответственных.** (Выполнено) — назначение задачи на конкретного члена семьи и фильтр «мои задачи» реализованы.

**Категории и сортировка покупок.** Частично реализовано: группировка товаров по категориям добавлена; сортировка под маршрут по магазину остаётся в планах.

**История и повторяющиеся элементы.** Часто покупаемые товары и повторяющиеся задачи/события (каждую неделю) — быстрый повторный ввод.

**Поиск и фильтры.** По мере роста списков нужен поиск по товарам/задачам и фильтр по дате/статусу.

## Безопасность и данные (Supabase)

**Аудит всех RLS-политик.** Пройти по каждой таблице и убедиться, что пользователь видит и меняет только данные своей семьи. Полезно написать набор тестов, которые проверяют доступ от лица «чужого» пользователя.

**Ограничение инвайт-кодов.** Сделать коды одноразовыми или со сроком действия, чтобы старый код нельзя было использовать бесконечно. Возможность отозвать код.

**Валидация на сервере.** Проверять длину/формат вводимых данных не только на клиенте, но и через ограничения БД (`CHECK`, `NOT NULL`, длина строк).

**Резервное копирование.** Включить автоматические бэкапы Postgres в Supabase и проверить восстановление.

## Производительность

**Виртуализация списков.** UI не должен рендерить тысячи DOM-элементов — добавить виртуализацию (react-virtual / react-window). Пагинация реализована для Покупок и Задач (загрузка порциями, "Загрузить ещё").

**История и повторяющиеся элементы.** Часто покупаемые товары и повторяющиеся задачи/события (каждую неделю) — быстрый повторный ввод.

**Поиск и фильтры.** По мере роста списков нужен поиск по товарам/задачам и фильтр по дате/статусу.

## Качество и надёжность кода

**Тесты.** Сейчас тестов нет. Минимум — несколько e2e-сценариев (вход, создание семьи, добавление товара, отметка «куплено») через Playwright, чтобы регрессии ловились автоматически.

**CI на GitHub.** GitHub Action, который на каждый push прогоняет `tsc`, `eslint` и сборку — не даст «сломанному» коду попасть на прод.

**Единый компонент состояний.** Загрузка / пусто / ошибка повторяются в каждом разделе — вынести в общий компонент, чтобы поведение было единообразным.

**Удалить неиспользуемое.** Компонент `BottomNav.tsx` существует, но нигде не импортируется — либо задействовать, либо удалить.

## UX-мелочи

PWA-манифест и иконка, чтобы приложение можно было «установить» на телефон и открывать с домашнего экрана как нативное.

Пустые состояния с подсказкой («Список пуст — добавьте первый товар») вместо просто пустого экрана.

Доступность: проверить контраст в тёмной теме, размеры кликабельных зон, навигацию с клавиатуры на ПК.

Лёгкая анимация при переключении разделов и отметке выполненного — приложение ощущается живее.
